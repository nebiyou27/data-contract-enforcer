# On-Call Runbook — Data Contract Enforcer

This runbook is for operators responding to alerts or failures in the data-contract-enforcer pipeline. It assumes you have the repo cloned and the package installed (`pip install -e .`).

---

## Quick orientation

| Component | What it does | Where to look |
|---|---|---|
| `contracts-run-all` | Validates all contracts in parallel | `batch.yaml`, `validation_reports/` |
| `contracts-evolution-gate` | Blocks breaking schema changes | GitHub Actions `evolution-gate` job |
| `violation_log/violations.jsonl` | Append-only violation record | Never delete; archive if too large |
| `schema_snapshots/baselines.json` | Drift detection baselines | Updated via `contracts-baseline promote` |
| `enforcer_report/report_data.json` | Latest health score + verdict | Generated by `contracts-report` |
| `quarantine/` | Records that failed schema validation | Reviewed via `contracts-quarantine` |

## Runbook entries

---

### P1 — Evolution gate blocked a deploy

**Symptom:** CI job `evolution-gate` fails with exit code 1. Message includes `BLOCK` and names a breaking field.

**What happened:** A PR removes or renames a field declared in `subscriptions.yaml` under `breaking_fields` for an active subscription. The gate prevents this from reaching `main` without a migration plan.

**Steps:**

1. Read the gate output to identify which field and which subscriber are affected:
   ```
   contracts-evolution-gate \
     --contract generated_contracts/<contract-id>.yaml \
     --proposed-ref HEAD \
     --current-ref origin/main
   ```

2. Decide: is this a legitimate schema change, or an accidental deletion?

   - **Accidental deletion** → restore the field in the PR and re-push.
   - **Intentional change** → proceed to step 3.

3. For an intentional change, open `contract_registry/subscriptions.yaml` and add a `migration_plan` to the subscription entry before the field is removed:
   ```yaml
   breaking_fields:
     - field: extracted_facts.confidence
       reason: "Week 4 depends on confidence semantics"
       migration_plan:
         strategy: dual-write
         cutover_date: "2026-05-01"
         owner: week3-team@trp.internal
   ```

4. Coordinate with the downstream subscriber team (contact is listed in the subscription entry).

5. Once the migration plan is registered and subscribers confirm readiness, re-run the PR.

**Do not bypass the gate** (`--no-verify` or removing the CI step) without explicit approval from the downstream subscriber. The gate exists to prevent silent breakage.

### P2 — Health score dropped below 70

**Symptom:** CI job `report` fails. Output says `Health score X is below minimum threshold (70)`.

**What happened:** One or more of: validation failures, new critical violations, or a schema breaking change pushed the composite health score below the operator-defined floor.

**Steps:**

1. Open the enforcer report to understand the breakdown:
   ```bash
   python -c "import json; d=json.load(open('enforcer_report/report_data.json')); print(json.dumps(d, indent=2))"
   ```
   Focus on: `data_health_score`, `verdict`, `violations.breakdown`, `validation.total_failed`.

2. If the score is low due to **validation failures** (`total_failed > 0`):
   - Open the failing validation report: `validation_reports/<contract-id>.json`
   - Find results where `status == "FAIL"` and read `message` and `column_name`
   - Fix the data or the contract depending on whether the data changed or the contract is stale

3. If the score is low due to **recorded violations**:
   - Read `violation_log/violations.jsonl`
   - Each entry has `check_type`, `violation_breakdown`, and `message`
   - For `range_breach`: the data produced an out-of-range value — investigate the producer
   - For `enum_violation`: a new undeclared enum value appeared — update the contract's enum list or reject the data
   - For `drift_mean`/`drift_variance`: the distribution shifted — decide if intentional and promote baselines if so

4. If the score is low due to **schema breaking changes**:
   - See [P1 — Evolution gate blocked a deploy](#p1--evolution-gate-blocked-a-deploy)

5. After fixing, re-run the validation suite and regenerate the report:
   ```bash
   contracts-run-all --batch batch.yaml
   contracts-report \
     --validation-reports validation_reports/*.json \
     --violation-log violation_log/violations.jsonl \
     --output enforcer_report/report_data.json
   ```

### P3 — Drift check FAIL on a known distribution shift

**Symptom:** `drift_mean` or `drift_variance` shows FAIL in a validation report. The shift is intentional (e.g., a new extraction model, a schema migration, a seasonal pattern).

**Steps:**

1. Confirm the shift is intentional by checking with the producer team.

2. Promote the current run's statistics as the new baseline:
   ```bash
   contracts-baseline promote --report validation_reports/<contract-id>.json
   ```
   This updates `schema_snapshots/baselines.json` for the affected contract.

3. Re-run validation to confirm drift checks now PASS:
   ```bash
   contracts-run \
     --contract generated_contracts/<contract-id>.yaml \
     --data outputs/<week>/data.jsonl \
     --output validation_reports/<contract-id>.json
   ```

4. Document the shift in `DOMAIN_NOTES.md` or a PR comment if it represents a meaningful change in data semantics.

**Never promote a baseline for a shift you haven't investigated.** Promoting an incorrect baseline resets the detection threshold and will miss future real drift.

### P4 — Quarantine records accumulating

**Symptom:** `quarantine/prompt_schema_violations.jsonl` is growing. Records are being quarantined faster than they are resolved.

**Steps:**

1. Review what's failing schema validation:
   ```bash
   contracts-quarantine review \
     --quarantine quarantine/prompt_schema_violations.jsonl \
     --samples 5
   ```
   The output groups violations by error type and shows sample records.

2. Determine the root cause:
   - **Missing required field** (`Missing required field: 'doc_id'`): the upstream producer changed its output format. Fix at the producer.
   - **Wrong type** (`Field 'processing_time_ms' must be number`): a type coercion issue upstream. Fix at the producer or update the schema.
   - **API key or credential in record** (`<redacted>` in the quarantine record): a sensitive field slipped into the payload. Escalate to the security team before requeuing.

3. Fix the records externally, then requeue:
   ```bash
   contracts-quarantine requeue \
     --quarantine quarantine/prompt_schema_violations.jsonl \
     --fixed-jsonl fixed_records.jsonl
   ```

4. After requeuing and confirming the fix, clear the quarantine:
   ```bash
   contracts-quarantine clear \
     --quarantine quarantine/prompt_schema_violations.jsonl \
     --yes
   ```
   The original file is archived to `quarantine/archive/` with a timestamp.

### P5 — Violation log is growing unexpectedly

**Symptom:** `violation_log/violations.jsonl` has many new entries. The `ai_extensions` check is writing violations on every run.

**Note:** The violation log is append-only. Do not delete entries.

**Steps:**

1. Read the most recent entries:
   ```bash
   tail -20 violation_log/violations.jsonl | python -m json.tool
   ```

2. Check `check_type`:
   - `llm_output_violation_rate` with `trend: increasing`: the verdict data quality is degrading. Check the Week 2 LLM output pipeline.
   - `range_breach`: a numeric field is producing out-of-range values. Find the producer and fix the data.
   - `enum_violation`: a new undeclared enum value appeared. Update the contract or reject the data.

3. For `llm_output_violation_rate` specifically:
   - Open `validation_reports/week2_ai_checks.json`
   - Read `violation_breakdown` to see which sub-violation (`missing_verdict`, `confidence_out_of_range`, etc.) is dominant
   - Fix at the source (the LLM prompt or output parser) and re-run AI checks:
     ```bash
     contracts-ai-checks \
       --data outputs/week2/verdicts.jsonl \
       --output validation_reports/week2_ai_checks.json
     ```

4. If the violation rate is expected (e.g., a test run with synthetic data), you can increase the threshold temporarily via env var rather than deleting log entries:
   ```bash
   ECE_VIOLATION_RATE_THRESHOLD=0.20 contracts-ai-checks ...
   ```

### P6 — Running out of disk space in schema_snapshots/

**Symptom:** `schema_snapshots/` is large. Each generator run appends a timestamped snapshot.

**Steps:**

1. Check how many snapshots exist per contract:
   ```bash
   ls schema_snapshots/week3-document-refinery-extractions/ | wc -l
   ```

2. Retain the two most recent snapshots and archive the rest:
   ```bash
   # Keep the last 2; move older ones to archive
   ls -t schema_snapshots/week3-document-refinery-extractions/ | tail -n +3 | \
     xargs -I{} mv schema_snapshots/week3-document-refinery-extractions/{} archive/snapshots/
   ```

3. Never delete `baselines.json` — it is the live drift reference. If it gets corrupted:
   ```bash
   contracts-baseline clear --all --yes
   # Then re-run and promote a clean baseline
   contracts-run-all --batch batch.yaml
   contracts-baseline promote --report validation_reports/week3_baseline.json
   ```

---

### P7 — CI passes but local run fails (environment mismatch)

**Symptom:** Validation passes in GitHub Actions but fails locally, or vice versa.

**Steps:**

1. Check that your local install matches CI:
   ```bash
   pip install -e .
   python -c "import contracts.config; print(contracts.config.config)"
   ```

2. Check for local env vars overriding thresholds:
   ```bash
   env | grep ECE_
   ```
   If any `ECE_*` variables are set locally and not in CI (or vice versa), that explains the discrepancy.

3. Check your local `schema_snapshots/baselines.json` — CI starts with a clean state, so drift checks behave differently if your local baselines are stale or absent:
   ```bash
   contracts-baseline list
   ```

4. Reset to a clean state:
   ```bash
   contracts-baseline clear --all --yes
   contracts-run-all --batch batch.yaml
   contracts-baseline promote --report validation_reports/week3_baseline.json
   ```

## Tunable thresholds (ECE_* env vars)

Set these to adjust enforcement without touching source code. All values are floats unless noted.

| Env var | Default | Effect |
|---|---|---|
| `ECE_DRIFT_Z_WARN` | `2.0` | z-score above which drift_mean fires WARN |
| `ECE_DRIFT_Z_FAIL` | `3.0` | z-score above which drift_mean fires FAIL |
| `ECE_DRIFT_VAR_WARN_HIGH` | `2.0` | stddev ratio above which drift_variance fires WARN |
| `ECE_DRIFT_VAR_WARN_LOW` | `0.25` | stddev ratio below which drift_variance fires WARN |
| `ECE_DRIFT_VAR_FAIL_HIGH` | `4.0` | stddev ratio above which drift_variance fires FAIL |
| `ECE_DRIFT_NULL_WARN_PP` | `0.05` | null-fraction growth (pp) for WARN |
| `ECE_DRIFT_NULL_FAIL_PP` | `0.20` | null-fraction growth (pp) for FAIL |
| `ECE_DRIFT_CARD_WARN_HIGH` | `2.0` | cardinality ratio spike for WARN |
| `ECE_DRIFT_CARD_WARN_LOW` | `0.5` | cardinality ratio collapse for WARN |
| `ECE_DRIFT_CARD_FAIL_HIGH` | `5.0` | cardinality ratio spike for FAIL |
| `ECE_VIOLATION_RATE_THRESHOLD` | `0.05` | LLM output violation rate that triggers a log entry |
| `ECE_LLM_TREND_DELTA` | `0.05` | Half-split delta for increasing/decreasing trend |
| `ECE_EMBED_WARN` | `0.1` | Cosine distance threshold for embedding drift WARN |
| `ECE_EMBED_FAIL` | `0.3` | Cosine distance threshold for embedding drift FAIL |
| `ECE_PROMPT_SCHEMA_FAIL_RATE` | `0.10` | Prompt schema violation rate for FAIL |
| `ECE_ENUM_CARDINALITY_LIMIT` | `10` (int) | Max distinct values for enum auto-inference |
| `ECE_MAX_BLAME_CANDIDATES` | `5` (int) | Top-N commits in blame chain |
| `ECE_GIT_LOG_LIMIT` | `20` (int) | Git log depth for blame |
| `ECE_CRITICAL_VIOLATION_PENALTY` | `20` (int) | Health score deduction per CRITICAL FAIL |
